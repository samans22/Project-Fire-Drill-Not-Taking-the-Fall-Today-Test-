## 二、核心问题诊断

### 问题 1：「第一局惊艳，第二局重复」

**症状**：20 个事件中 15 个是 `projectTags: ["*"]` 通用事件。无论选 ESG、IPO 还是峰会，85% 的事件都一样。玩家选不同项目期望"不同体验"，实际得到"换了个标题的同一局"。

**根因**：项目差异化仅体现在事件抽取的标签过滤，而非叙事、数值、策略的深层差异。

### 问题 2：「选择像做选择题，不像做策略」

**症状**：当前每个选项的 effects 是静态的——选了 A 就是固定的 {time:-2, risk:-2}。玩家看不到"我上次选了 A 导致后续被卡住，这次试试 B"的策略演化。

**根因**：连锁事件（E016-E020）是最有策略感的系统，但只有 5 条链，且全部是单步触发。缺少多步因果链和状态依赖的选项。

### 问题 3：「3 分钟一局太短，没有"再来一局"的冲动」

**症状**：单局 3-5 分钟，通关后看个结局就结束了。没有局外进度、没有解锁内容、没有"这次我要试试不同打法"的动力。

**根因**：纯局内设计——每次都是 clean slate，缺少 meta-progression。

### 问题 4：「项目专属事件太少」

**症状**：

| 项目 | 专属事件数 | 可遇到的总事件 |
|------|-----------|---------------|
| ESG 年报 | 2 (E001, E006) | ~17 |
| IPO 路演 | 1 (E010) | ~16 |
| 行业峰会 | 2 (E002, E017) | ~17 |

实际上每个项目的专属事件在单局中只能触发 1-2 次。

### 🔴 P1 — 策略深度：多步因果 + 事件组合

#### 3.3 二级连锁系统

**当前**：flag → 1 个连锁事件（单步）  
**目标**：flag → 连锁事件中的选择再设 flag → 更深层后果（多步）

```
示例：ESG 报告的因果树

E001 (碳减排数据缺口)
 ├─ A: 追要原始数据 → 无标记，但后续可能触发「数据部门投诉你太强势」
 ├─ B: 定性表述先顶上 → data_missing → E016 (合规追问)
 │   ├─ A: 亡羊补牢 → 风险 -2
 │   └─ B: 争取时间 → 再设 tentative_data → 结局扣分
 └─ C: 参考行业数据 → cross_reference → 可能触发「竞品注意到了你们的参考」
```

#### 3.4 条件选项（同一事件内根据状态不同出现不同选项）

**示例**：在 E005（预算报销被卡）中：

```json
{
  "text": "找林总特批",
  "effects": { "time": -1, "budget": 1, "satisfaction": -1, "risk": 0 },
  "requiresFlags": []  // 总是出现
},
{
  "text": "拿之前的合规记录证明你的严谨",
  "effects": { "time": 0, "budget": 1, "satisfaction": 1, "risk": -1 },
  "requiresFlags": ["compliance_clean"]  // 只有之前合规全通过的玩家才能选
}
```

#### 3.5 权衡加深：二阶段选择

某些关键事件不是一次选择就结束，而是「选方向 → 看反馈 → 再选执行方式」：

```
E010 (招股书数据调整):
  第一阶段：选择策略方向
    A. 自己加班改 → 解锁 3 个子选项（怎么加班）
    B. 找外部咨询 → 解锁 2 个子选项（找哪家）
    C. 折中方案 → 直接结束

  第二阶段的子选项影响具体数值：
    选了 A：
      A1. 通宵一次搞定 (时间-3, 满意度+1)
      A2. 分两天改 (时间-1, 风险+1, 每天推进1天)
      A3. 拉阿哲一起加班 (时间-2, 满意度+2, 但消耗阿哲好感)
```

### 🟡 P2 — 重玩价值：Meta-Progression + 内容解锁

#### 3.6 局外解锁系统

**当前**：每次新游戏都是完全干净的体验  
**目标**：增加「成就 + 解锁」驱动重玩

| 系统 | 说明 | 示例 |
|------|------|------|
| **成就系统** | 约 15-20 个成就，触发后记录 | 「死也不改」：5 次选择"最终版"相关选项 / 「守夜人」：触发 E014-C 让小许守夜 / 「铁算盘」：3 局不超预算 |
| **结局图鉴** | 11 个结局，收集全部有彩蛋 | 主界面显示 `结局收集: 3/11` |
| **角色好感** | 隐藏的 NPC 好感度（不显示数值） | 总选阿哲的方案 → 他在 E018 可能主动帮你修 / 总让实习生善后 → 结局里他辞职 |
| **项目履历** | 每局结束生成"年度考核表" | 可直接截图分享到社交媒体 |

#### 3.7 每日挑战 / 种子模式

- 每天生成一个固定种子 → 所有玩家遇到相同的事件序列
- 排行榜：看谁的结局评分最高
- 分享种子 → 朋友可以挑战"你的这一天"

### 🟡 P3 — 叙事厚度：事件变体 + 角色弧线

#### 3.8 事件变体

同一个事件根据上下文有不同的 flavor text，不改变 effects 但提升沉浸感：

```
E012 (PPT文件名混乱):
  如果之前 E007 (实习生发错邮件) 选了让小许自己处理：
    → 小许说"上次你让我自己处理，这次我要证明自己"
    → 选项 C 的反馈会不同："小许帮你整理了版本，但他把最新的删了"
```

#### 3.9 角色小故事线

8 个角色各有一条跨事件的微型叙事线（2-3 个事件串联）：

| 角色 | 故事线 | 触发条件 |
|------|--------|----------|
| 技术阿哲 | 「996 还是 955」— 你总压榨他还是尊重他的时间 | E008 + E018 + 新事件 |
| 实习生小许 | 「菜鸟成长记」— 你放养还是培养他 | E007 + E014 + 新事件 |
| 合规陈老师 | 「从敌人到盟友」— 你尊重合规还是总绕过去 | E009 + E020 + E015 |
| 供应商王姐 | 「已读不回的秘密」— 她背后有什么故事 | E002 + E006 + 新事件 |

## 六、数值平衡参考

### 6.1 当前选项 effects 分布

```
时间变化:  -3 (5次)  -2 (13次)  -1 (20次)  0 (16次)  +1 (2次)
预算变化:  -3 (1次)   -2 (2次)   -1 (7次)   0 (44次)  +1 (2次)
满意度:    -2 (1次)   -1 (8次)   0 (39次)   +1 (6次)  +2 (2次)
风险变化:  -3 (3次)   -2 (10次)  -1 (13次)  0 (17次)  +1 (11次)  +2 (8次)
```

### 6.2 一局约 5-7 次选择的净效果预测

以中等难度项目（如 ESG 初始时 7）计算，5-7 次选择后：
- 时间：平均净 -1.5/次 → 剩余约 2-3（接近警戒线）
- 预算：平均净 -0.3/次 → 剩余约 8（充裕）
- 满意度：平均净 +0.1/次 → 基本不变
- 风险：平均净 +0.5/次 → 累积 3-4（中等）

**问题**：预算几乎是"死数值"——只有 1 个事件的 1 个选项扣 3 预算，其余事件几乎不碰预算。

**建议**：让预算在更多事件中出现，尤其是专属事件中可以设计为"要么花时间要么花钱"的经典 tradeoff。

### 6.3 主题联动设计

当前 19 个主题只是顶部栏的文字。可以赋予实际效果：

| 主题 | 实际 effect |
|------|------------|
| 项目预算被砍 30% | 初始预算 -3 |
| 客户突然加需求 | 初始时间 -2 |
| 供应商集体掉链子 | 前 3 个事件风险 +1 |
| 最后一个工作日 | maxDays 从 10 改为 3（短局） |
| 周五下午四点整 | 时间初始 -1，但小许会主动帮忙 |

## 七、总结

### 当前优势
- 🎨 **伪装 UI**：企业后台风格是最大亮点，与市面上所有文字游戏拉开差距
- ✍️ **叙事对味**：反馈文本精准踩中职场人的痛点和笑点
- 🧱 **架构干净**：数据驱动 + 模块化，扩展成本低
- 📱 **分享友好**：单文件 HTML 天然适合微信/QQ 传播

### 核心瓶颈
- 📉 **内容量不足**：20 事件 → 需要至少 45 事件才能支撑 10 分钟+ 的有深度体验
- 🔄 **项目无差异**：3 个项目换皮 → 需要 60%+ 专属事件
- 🎲 **重玩无动力**：无局外系统 → 需要成就/图鉴/好感度

### 建议的 MVP 交付节奏

```
Week 1-2:  新事件写作（15-19 个专属事件）          → v0.6
Week 3-4:  策略系统（二阶段 + 条件选项 + 多步链）   → v0.7
Week 5-6:  局外系统（成就 + 图鉴 + 好感度）         → v0.8
Week 7-8:  叙事润色（变体 + 角色弧线）             → v0.9
Week 9-10: 打磨测试 + 正式发布                      → v1.0
```

总工时估算：**约 10 周**（单人业余时间），可压缩至 **5-6 周**（全职）。
